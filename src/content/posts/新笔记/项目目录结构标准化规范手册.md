## 1 为什么需要多套规范

不同项目的核心产物不同，不能强行套同一套目录。

| 项目类型      | 核心产物                       | 推荐规范                |
| --------- | -------------------------- | ------------------- |
| 探索型生信分析项目 | 图表、统计结果、中间分析对象、论文方法        | [[探索型生信分析项目管理规范手册]] |
| 产品型/软件型项目 | 可运行程序、数据库、API、网站、长期维护的数据产品 | [[产品型项目管理规范手册]]     |

判断标准：

> 如果项目主要回答科学问题，用“探索型生信分析项目”规范。  
> 如果项目主要交付可运行、可维护的软件或数据产品，用“产品型项目规范”。

## 2 两类项目的核心区别

| 维度 | 探索型生信分析项目 | 产品型/软件型项目 |
|---|---|---|
| 主线目录 | `code/` | `src/` |
| 辅助脚本 | `utils/workflows/` 或 `code/` 中的步骤脚本 | `scripts/` |
| 数据 | `data/raw/`, `data/processed/` | `data/raw/`, `data/processed/`, `data/database/` |
| 输出 | `results/figs/`, `results/tables/` | 构建产物、数据库、网页、API 输出 |
| 文档 | 分析计划、方法草稿、运行记录 | 使用说明、开发说明、部署说明、API 文档 |
| 测试 | 可选，常用于结果检查 | 推荐必备，放在 `tests/` |

## 3 命名选择

| 名称 | 适合场景 |
|---|---|
| `code/` | 主分析流程脚本，适合生信探索分析 |
| `src/` | 软件主体源码，适合包、网站、API、数据库系统 |
| `scripts/` | 命令行入口、批处理、下载、构建、部署脚本 |
| `utils/` | 可复用函数、调色板、marker、模板、公共资源 |

一句话：

> 分析项目用 `code/` 做主线；软件项目用 `src/` 做主体；批处理任务放 `scripts/`；通用能力放 `utils/`。

## 4 推荐使用方式

新建项目前先回答三个问题：

1. 这个项目的最终交付物是什么？
2. 代码是为了完成一次分析，还是为了长期运行和维护？
3. 结果主要是图表/表格，还是程序/数据库/网站/API？

对应选择：
```text
科学问题、论文复现、单细胞/RNA-seq/WGCNA/DEG/GSEA
=> 使用 [[探索型生信分析项目]]

数据库、知识库、网站、API、工具包、长期更新数据产品
=> 使用 [[产品型项目规范]]
```

## 5 最小共识
无论哪类项目，都建议保留：
```text
ProjectName/
├── README.md
├── data/
└── docs/
```
其中：
- `README.md`：项目入口说明。
- `data/`：项目使用或生成的数据。
- `docs/`：人类可读的说明、计划、记录和维护文档。
